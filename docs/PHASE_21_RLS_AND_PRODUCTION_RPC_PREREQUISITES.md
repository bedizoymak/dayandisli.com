## RLS Audit Findings

The original ERP schema created broad authenticated policies dynamically. Later governance migrations replaced many of those policies with membership-based policies, but `work_order_operations` was not given a durable replacement.

The generated membership predicates use unqualified target columns inside a subquery that aliases `company_memberships` as `cm`. PostgreSQL resolves expressions such as `cm.company_id = company_id` to the inner relation when possible. Effective policy inspection showed this as:

```sql
cm.company_id = cm.company_id
```

The same risk applies to branch predicates. These expressions do not enforce the intended tenant comparison and must be replaced with explicit target-table qualification.

The audit also found legacy allowance for rows whose `company_id` is null. This behavior is retained in the drafts to avoid an unreviewed compatibility change, but it remains a security and data-model risk.

## work_order_operations Policy Design

Access is derived through the parent `work_orders` row:

1. Match `work_order_operations.work_order_id` to `work_orders.id`.
2. Permit legacy parent rows with null `company_id`.
3. Otherwise require an active `company_memberships` row for the authenticated email and parent company.
4. Permit company-wide membership when `company_memberships.branch_id` is null.
5. Permit branch-scoped membership only when it exactly matches a non-null parent branch.

The review draft defines authenticated `SELECT`, `INSERT`, and `UPDATE` policies. It intentionally grants nothing to `anon`. Table privileges and existing overlapping policies must be audited before packaging the final migration.

This design is stricter than the old generated branch expression: a branch-scoped member cannot access a parent row with a null branch. That tightening requires staging verification.

## Tenant Predicate Correction Plan

The correction draft uses explicit aliases and target-table qualification for the three tables directly involved in the proposed production workflows:

- `sales_orders`
- `work_orders`
- `erp_audit_logs`

It also includes a catalog query to locate suspicious effective policy expressions. The draft is deliberately not a repository-wide policy rewrite. Every table generated by the Phase 25, 26, 29, and 30 policy loops requires individual review for:

- intended null-company behavior;
- company-wide versus branch-scoped access;
- required command coverage;
- coexistence with admin or service policies;
- table-specific ownership paths;
- existing grants to authenticated and anonymous roles.

## work_orders.sales_order_id Uniqueness Plan

The production workflow requires one work order per non-null sales order. The draft:

1. Reports duplicate `sales_order_id` groups.
2. Raises an exception when any duplicate group exists.
3. Creates a partial unique index only for non-null values.

No duplicate cleanup is included. `CREATE UNIQUE INDEX CONCURRENTLY` must run outside a transaction, so final migration packaging must account for the migration runner's transaction behavior.

## Static Verifier

`scripts/verify-production-prereq-sql.mjs` checks that:

- all prerequisite drafts exist;
- every draft is marked `DRAFT ONLY`;
- no draft uses elevated function execution;
- no draft grants execution to `anon`;
- the operation policy resolves tenant access through `work_orders`;
- the predicate correction draft omits the known ambiguous expression;
- the uniqueness draft checks duplicates before creating a partial unique index.

## Remaining Risks

- Review-only SQL has not been applied or integration-tested.
- Existing overlapping policies and table grants may broaden access.
- Null-company compatibility remains an intentional legacy bypass.
- Branch semantics may affect existing null-branch work orders.
- The full set of generated tenant policies still needs table-by-table correction.
- Existing duplicate sales-order links would block unique index creation.
- Concurrent index creation requires an execution path outside a transaction.

## Next Recommended Phase

Review the drafts, inventory every affected effective policy and grant in an isolated local database, then package narrowly scoped migrations with local integration tests before any staging rollout.
